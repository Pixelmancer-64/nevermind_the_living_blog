# Por que fazer estágio?

Fazer estágio pode ser uma boa ideia para você que ainda não sabe exatamente o que quer fazer da vida, principalmente se você for um estudante. O estágio que me refiro aqui no caso é direcionado para estudantes do ensino médio/univerdade que não possuem a perspectiva de trabalhar por 8 horas diárias. No meu caso, o meu estágio foi executado por um perído de aproximadamente 7 meses, onde metade 2/3 das minhas 400 horas foram cumpridas em expedientes de 3+ horas por dia, 3 dias por semana, e o resto em expedientes de 6+ horas, 5 dias por dia. 

O meu ensino médio também contempla o ensino técnico de Informática, o que significava que em duas dias da semana eu possuia aulas até às 18:00, imposibilitando a realização do estágio durante a semana toda.

Bem, além de descobrir uma área que você tenha interesse, o estágio é importantíssimo para aprender a como se portar no local de trabalho e como aplicar o teórico no prático. Eu por exemplo já estudo por conta própria, sem depender do currículo da minha escola, e desenvolvo alguns projetos no meu tempo livre e, posso falar por experiência própria, que nenhum curso na Udemy ou vídeo no Youtube vai te preparar para como é realmente trabalhar com prazos e com demandas, ainda mais no setor público.

## Python

Eu já conhecia Python antes do estágio e já utilizava frequentemente a linguagem para fazer alguns projetos para geração de [fractais](https://github.com/Pixelmancer-64/PythonIFS) e [processamento de imagens](https://github.com/Pixelmancer-64/imageTools), mas nunca tinha usado com tal profundidade até esse momento.

Utilizamos o framework [Django](https://www.djangoproject.com/) em todos os sistemas desenvolvidos e, até então, eu desconhecia o uso da linguagem Python no campo do desenvolvimento web. Apesar do minha familiaridade com a linguagem, que vai além do CEFET/RJ (essa linguagem não foi lecionada durante minha estadia na escola), eu possuia maior confiança com Javascript, Rust, C++ e PHP.

Então, logo no primeiro dia, já enfrentei o meu primeiro *choque de cultura*, conhecendo um novo mundo que me aguardava, um mundo esse que era familiar, mas mesmo assim diferente.

## MariaDB / SQL

O MySQL é o gerenciador de banco de dados relacionais mais popular do mundo, mas não faz muito tempo ele foi adquirido e incorporado pela Oracle, o que restringiu em parte o uso dessa ferramenta para grandes projetos. O [MariaDB](https://mariadb.org/) é a alternativa [open-source](https://pt.wikipedia.org/wiki/Software_de_c%C3%B3digo_aberto) do MySQL Server e foi exatamente essa tecnologia que é usada no servidor de banco de dados dos nossos sistemas. 

A diferença entre MySQL e MariaDB é pontual, basicamente idênticos. Vale ressaltar que a linguagem continua a mesma, consultas ao banco de dados são feitas por SQL. Na escola temos sim algumas matérias sobre diagramação, SQL e configuração do servidor MySQL que me ajudaram na hora de trabalhar com alguns projetos.

Eu gosto muito de banco de dados e tenho certo conhecimento sobre SQL, o que me permitiu a transitar algums sistemas, que utilizavam Python para consultar nossos bancos de dados, para consultas utilizando SQL. Essa mudança foi muito bem-vinda pelo ganho de desempenho e espaço no servidor; algumas funcionalidades de alguns sites levavam cerca de meia hora para carregarem, e agora levavam questão de segundos.

## Django

Falando mais especialmente do Django, é uma das tecnologias que mais me divertir a aprender. Ela utiliza o modelo MVC, mas apesar do modelo ele altera o nome das "partes " do projeto em si, e possui um fluxo de trabalho bem parecido com outras frameworks.

Django foi mais uma das coisas que aprendi diretamente com a prático e a leitura da sua[documentação](https://docs.djangoproject.com/en/4.1/). Já estou acostumado a aprender dessa maneira, como foi o caso do Typescript e Rust, mas foi impressionantemente fácil compreender e me adaptar à ferramenta.

## Shell script

Utilizei muito brevemente mas vale a menção. No servidor de banco de dados, utilizavam uma rotina de backups que era executada todo dia, durante a noite, para gerar um backup para cada sistema instalado no servidor. O código não era muito conveniente então aprendi um pouquinho de Shell script para alterar o comando já lá existente e melhorar a rotina de backups do servidor.

## Apache server

Apesar da introdução que o ensino técnico ofereceu a respeito da configuração e manutenção de servidores apache, o estágio proporcionou uma visão completamente diferente do processo de criação de arquivos de configuração e emissão de certificados ssl utilizando o certbot. Acessar o servidor através do SSH foi inicialmente um desafio, devido a ausência de interface e limitações de formas mais modernas de editores de texto. Utilizar o servidor remotamente proporcionou um ganho de conhecimento imenso, abragendo questões como: SSH, paths absolutos e relativos, permissões e responsabilidade ao utilizar o superusuário, versionamento utilizando o github e utilização de comandos do terminal linux para manipulação de arquivos e diretórios.

## Github

O Github é uma ferramenta essêncial para todo programador. De forma geral, pode ser chamado de banco de dados para repositórios de projetos, mas também pode ser considerado como uma rede social para programadores. Seu uso se resume a armazenar na nuvem todo o código desenvolvido localmente por várias pessoas em um só lugar, facilitando a manutenção de grandes codebases gerenciadas por diversos programadores. Apesar do conhecimento já extenso que possuia antes do estágio (já utilizava o github diariamente por 2 anos), trabalhar em junção com outras pessoas e compreender como resolver conflitos no momento de juntar duas branchs foram habilidades que adquiri durante meu período durante a secretaria.

## Trello

O meu supervisor, o programdor Luis Eduardo "e-salarini" Cordeiro Salarini, é um ávido usuário de post-it. Pensando no impacto ambiental e no custo elevado desse material tão essencial para a vida organizada no espaço de trabalho, ele introduziu o método "Kanban" de organização e passou a utilizar o site Trello para gerenciar os projetos desenvolvidos na secretaria. Porém, após o uso da plataforma por 3 semanas, optamos por retornar ao uso de post-its devido a estrturua do Trello, que muitas vezes passava a ser a parte mais cansativa do trabalho; muitas vezes nos encontrávamos gastando mais tempo classificando e explicando as tarefas que realizamos do que efetivamente fazendo elas. O retorno ao meio físico do método Kanban foi bem recebido por mim, meu supervisor e a única outra programadora e membro de equipe, Ana Luísa Ferro 

## Asana

Após utilizar os post-its por 2 meses, um servidor público da secretaria de TI nos introduziu a plataforma Asana, que mudou completamente a forma como gerenciamos nossos projetos. Seguindo também a ideia do Kanban, o Asana possui várias colunas que armazenam as tarefas dependendo da sua categoria e marcar-las como concluidas é tão simples quanto clicar um botão. Foi uma mudança bem-vinda por nós três e agilizou a criação, acompanhamento e conclusão das tarefas, sendo até uma ferramenta útil para conferir quais tarefas e exigências já haviam sido cumpridas.

## Figma

O Figma é utilizado para a criação de wireframes e desenvolvimento do design de páginas web e outros elementos gráficos. No nosso caso, o Figma foi utilização unicamente para o desenvolvimento web, mas o domínio da ferramenta também auxilia em outras áreas de design.
